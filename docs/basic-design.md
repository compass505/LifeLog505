# LifeLog505 基本設計書 第1版

## 1. 基本設計の目的

本書は、要件定義書で定めた日記・ライフログアプリ「LifeLog505」の内容をもとに、アプリの画面構成、画面遷移、データ構造、AI処理、認証、検索機能などの基本的な設計方針を整理するものである。

本設計書は、今後の詳細設計およびCodexを用いた実装作業の前提資料として利用する。

---

## 2. アプリ名

本アプリの名称は、現時点では以下とする。

text LifeLog505 

---

## 3. アプリの基本方針

LifeLog505は、ユーザーがAIと友達のように楽しく自然に会話することで、日々の出来事や感情、食事、運動などを記録できる日記・ライフログアプリである。

本アプリでは、ユーザーに「日記を書いている」「記録している」と強く意識させるのではなく、友達と話しているような自然な体験を重視する。

そのため、AIチャットでは以下を重視する。

- 友達と楽しく話しているような体験
- 自然な共感
- 軽い雑談
- 話しやすさ
- 質問しすぎないこと
- 日記作成を前面に出しすぎないこと

AIは、会話中に以下のような表現を原則使用しない。

text 日記に残しておくね 記録しておくね メモしておくね 

日記本文の生成や食事・運動データの整理は、ユーザーに見えない裏側の処理として行う。

---

## 4. システム構成

本アプリは、以下の構成で開発する。

text React + Supabase + Supabase Edge Functions + OpenAI API 

### 4.1 フロントエンド

フロントエンドにはReactを使用する。

Reactは以下を担当する。

- 画面表示
- チャットUI
- 日記一覧表示
- 日記詳細表示
- 検索画面
- ログイン画面
- 新規登録画面
- 設定画面
- ユーザー操作の受付
- Supabaseとの通信

### 4.2 バックエンド・データベース

バックエンドおよびデータベースにはSupabaseを使用する。

Supabaseは以下を担当する。

- ユーザー認証
- 日記データの保存
- 会話ログの保存
- 食事・運動などの構造化データ保存
- ユーザーごとのデータ分離
- 将来的なベクトル検索

### 4.3 AI処理

AI処理にはOpenAI APIを使用する。

OpenAI APIはフロントエンドから直接呼び出さず、Supabase Edge Functionsなどのサーバー側処理を経由して実行する。

これにより、OpenAI APIキーをフロントエンドに公開しない設計とする。

## 8. 日記生成方針

### 8.1 日記本文はチャットのたびに生成しない

本アプリでは、チャットのたびに日記本文を毎回生成・更新しない。

チャット時は、主に以下を行う。

- AIとの自然な会話
- 会話ログの保存
- 必要に応じた構造化データの保存
- 必要に応じた日付判定

日記本文は、1日の終わりにまとめて生成する。

---

### 8.2 MVPでの日記生成タイミング

MVPでは、自動生成ではなく、以下のボタンによる手動生成を基本とする。

text 今日の日記を作成する 

ユーザーがこのボタンを押すと、その日の会話ログや構造化データをもとに、AIが日記本文を生成する。

---

### 8.3 将来的な日記生成タイミング

将来的には、以下のような自動生成を検討する。

- 毎日夜に自動生成する
- 翌日初回アクセス時に、前日の日記が未生成なら自動生成する
- ユーザーが任意のタイミングで再生成する

---

### 8.4 日記本文の文体

日記本文は、普通の日記として自然に読める文章とする。

#### 方針

- 300〜600字程度
- 自然な文章
- 事実と感情を含める
- ポジティブに盛りすぎない
- ユーザー本人が後から読んでも違和感がない文体
- 箇条書きではなく文章形式

---

## 9. 過去日の日記更新方針

### 9.1 過去日の内容は即時更新しない

チャット中に過去日の日記に関する内容が出た場合でも、原則としてその場で日記本文を即時更新しない。

たとえば、ユーザーが以下のように入力した場合である。

text 昨日の研修で発表したことも書いておきたい 

この場合、すぐに昨日の日記本文を書き換えるのではなく、対象日付と追記内容を未反映データとして保存する。

---

### 9.2 変更がある日付のみまとめて更新する

日記生成・更新処理のタイミングで、変更がある日付のみAIに渡して日記本文を更新する。

これにより、AI呼び出し回数を抑えつつ、日記本文を自然な文章として整える。

---

### 9.3 日付が曖昧な場合

日付が曖昧な場合は、AIが自然に確認する。

例：

text この前の飲み会楽しかった 

この場合、対象日付が特定できないため、AIは以下のように確認する。

text それって先週くらいの飲み会のこと？ 

日付が特定できるまでは、過去日記への反映は行わない。

---

## 10. AIチャット設計

### 10.1 AIの基本的な振る舞い

AIは、日記作成担当ではなく、友達のような話し相手として振る舞う。

#### 重視すること

- 楽しく話す
- 共感する
- 軽く反応する
- 話を広げる
- 質問しすぎない
- ユーザーの話を否定しない
- 日記作成を前面に出さない

---

### 10.2 避ける表現

AIは以下のような表現を原則使用しない。

text 日記に残しておくね 記録しておくね メモしておくね 保存しておくね 

理由は、ユーザーに「記録されている」という意識を強く持たせず、友達と話しているような自然な体験を優先するためである。

---

### 10.3 望ましい返答例

#### 例1

ユーザー：

text 今日研修で結構疲れた 

AI：

text おつかれ！研修って座ってるだけでも意外と体力持っていかれるよね。 今日は頭も体もけっこう使った感じ？ 

#### 例2

ユーザー：

text 昼はパスタサラダ食べた 

AI：

text いいじゃん、ちゃんと昼で調整できてるね。 パスタサラダって軽すぎないし、続けやすいのが強いよね。 

#### 例3

ユーザー：

text 夜ジムでスクワットやった 

AI：

text えらい！夜にジム行ってスクワットできるの普通に強い。 疲れててもそこまでできたなら、かなり頑張った日だね。 

---

## 11. AIに渡す情報の方針

AIに毎回すべての情報を渡すことはしない。

毎回渡す情報は、必要最小限に絞る。

### 11.1 チャット時に渡す情報

チャット時には、主に以下をAIに渡す。

text 今回のユーザー入力 直近の会話数件 ユーザーの重要プロフィール 最近の傾向 当日の簡易サマリー AIの振る舞い指示 

### 11.2 毎回渡さない情報

以下の情報は毎回渡さない。

text 過去の日記全文 全ての会話ログ 全ての食事履歴 全ての運動履歴 

必要になった場合のみ、検索や条件指定によって関連情報を取得し、AIに渡す。

## 14. 認証設計

認証にはSupabase Authを使用する。

### 14.1 認証方式

MVPではメールアドレスとパスワードによる認証を利用する。

### 14.2 認証後の動作

ログイン成功後、チャット画面へ遷移する。

ログアウト後、ログイン画面へ戻る。

### 14.3 データアクセス制御

各テーブルにはuser_idを持たせる。

ログイン中ユーザーは、自分のuser_idに紐づくデータのみ閲覧・作成・更新・削除できる。

SupabaseのRow Level Securityを利用し、ユーザーごとのデータ分離を行う。

## 15. AI処理設計

AI処理は以下の3種類に分ける。

text 1. チャット返答生成 2. 構造化データ抽出 3. 日記本文生成・更新 

---

### 15.1 チャット返答生成

ユーザーの入力に対して、AIが友達のような自然な口調で返答する。

チャット返答生成では、日記本文を毎回生成しない。

---

### 15.2 構造化データ抽出

AIは会話内容から、必要に応じて食事・運動・感情・タグなどのデータを抽出する。

#### 抽出対象

text meals exercises mood tags target_date needs_diary_update 

#### 食事データ例

json {   "meal_type": "lunch",   "description": "パスタサラダ",   "calories": null,   "memo": "会社の昼食" } 

#### 運動データ例

json {   "exercise_name": "スクワット",   "sets": 3,   "reps": 10,   "weight": 80,   "duration_minutes": null,   "memo": "ジムで実施" } 

---

### 15.3 日記本文生成・更新

日記本文は、1日の終わりにその日の会話ログと構造化データをもとに生成する。

過去日の日記についても、変更候補がある場合のみ更新する。

#### 生成内容

text title body mood summary tags 

---

## 16. AI処理の流れ

### 16.1 チャット送信時

チャット送信時の基本的な処理の流れは以下とする。

text 1. ユーザーがチャットを送信する 2. フロントエンドがSupabase Edge Functionにリクエストを送る 3. Edge Functionがログインユーザーを確認する 4. 直近の会話ログ、ユーザープロフィール、当日の簡易サマリーを取得する 5. OpenAI APIに以下を渡す    - 今回のユーザー入力    - 直近の会話ログ    - ユーザーの重要プロフィール    - 最近の傾向    - 当日の簡易サマリー    - AIの振る舞い指示 6. OpenAI APIから返答と必要な抽出データを受け取る 7. ユーザーメッセージを保存する 8. AIメッセージを保存する 9. 必要に応じて食事・運動データを保存する 10. 過去日への反映が必要な場合はdiary_update_requestsに保存する 11. フロントエンドにAI返答を返す 12. チャット画面に表示する 

---

### 16.2 今日の日記を作成する時

text 1. ユーザーが「今日の日記を作成する」ボタンを押す 2. フロントエンドがEdge Functionにリクエストを送る 3. Edge Functionが対象日付の会話ログを取得する 4. 対象日付の食事・運動データを取得する 5. 対象日付の未反映の更新候補を取得する 6. OpenAI APIに日記生成を依頼する 7. AIが日記タイトル、本文、感情、要約、タグを生成する 8. diariesテーブルに保存または更新する 9. 関連するdiary_update_requestsをappliedに変更する 10. フロントエンドに生成結果を返す 11. 日記詳細画面に表示する 

---

## 17. API設計方針

MVPでは、フロントエンドから直接Supabaseを利用できる処理と、Edge Functionを経由する処理を分ける。

### 17.1 フロントエンドからSupabaseを直接利用する処理

text ログイン 新規登録 ログアウト 日記一覧取得 日記詳細取得 日記編集 日記削除 検索 プロフィール取得 プロフィール更新 

### 17.2 Edge Functionを経由する処理

text AIチャット送信 構造化データ抽出 今日の日記生成 過去日の日記更新 

OpenAI APIを利用する処理は、必ずEdge Functionを経由する。

### 18.1 検索対象

text diaries.title diaries.body diaries.summary tags.name

## 19. 権限・セキュリティ設計

### 19.1 ユーザーデータの分離

全ての主要テーブルにuser_idを持たせる。

ユーザーは自分のデータのみ操作できる。

### 19.2 APIキーの保護

OpenAI APIキーは環境変数として管理する。

フロントエンドにはAPIキーを含めない。

### 19.3 RLS

Supabase Row Level Securityを利用する。

各テーブルに対して、以下の方針でポリシーを設定する。

text select: 自分のデータのみ取得可能 insert: 自分のデータのみ作成可能 update: 自分のデータのみ更新可能 delete: 自分のデータのみ削除可能
